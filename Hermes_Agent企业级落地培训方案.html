AI Skill Enablement Blueprint

Hermes Agent
企业级落地培训方案

在 Ubuntu 上搭建本地 Agent 中枢，以 Cursor CLI 作为推理后端，通过 OpenAI 兼容网关接入 Hermes；再连接企业微信，并建立“Agent 写 PRD → 一键发布 → 团队在线阅读”的文档闭环。

适用系统：Ubuntu 22.04 / 24.04 LTS 培训时长：4–6 小时 部署形态：单机 PoC → 内网生产化 文档版本：2026-07-17

Module 01培训目标与最终交付

这不是一堂“看完就算”的概念课。培训结束时，学员应当拥有一套能运行、能演示、能复用、能审计的 Agent 工作台。

1

本地 Agent 中枢

Hermes Agent 在 Ubuntu 常驻运行，具备记忆、技能、工具与 OpenAI 兼容 API。

2

模型接入网关

Cursor Agent CLI 经 agent-cli-to-api 转换为本机 OpenAI 兼容接口。

3

企业协作入口

员工可在企业微信中与 Hermes 双向对话，不必暴露公网 Webhook。

4

PRD 共享站

PRD 使用 Git 版本管理，由 GitHub Actions 发布到 GitHub Pages。

5

一键发布 Skill

Agent 按模板写 PRD、校验、提交并触发发布，返回共享链接。

6

安全运维基线

接口仅监听回环地址，密钥隔离，用户白名单，服务可观测、可回滚。

一句话业务价值 把“一个人会用 AI”升级为“团队拥有统一的 AI 生产线”：需求在企业微信进入，Hermes 执行，PRD 在 Git 中沉淀，网页自动发布。

Module 02总体架构：两层 API、三个入口

推荐部署拓扑：Cursor CLI 是推理后端，Hermes 是 Agent 中枢，企业微信与 PRD 站是业务入口和交付出口。

为什么是“两层 API”

接口地址作用谁调用
模型适配 API127.0.0.1:8000/v1把 Cursor Agent CLI 包装成 OpenAI 兼容模型接口Hermes
Agent 能力 API127.0.0.1:8642/v1暴露 Hermes 的记忆、Skill 和工具执行能力自建应用、内网前端、扩展机器人

企业微信优先直接使用 Hermes 原生 WeCom 适配器。只有需要自定义审批、工单或业务系统编排时，才由自建服务调用 :8642/v1。

Module 03上线前先讲清楚：产品边界与合规风险

重要：这不是 Cursor 官方模型 API Cursor CLI 与 User API Key 是官方能力；agent-cli-to-api 是第三方开源适配器。它把 CLI 包装成 HTTP API，不等于 Cursor 官方承诺的稳定服务。用于团队共享、生产流量或对外服务前，必须核对 Cursor 当前订阅条款、组织政策、并发与配额规则。

适合

  • 个人或小团队 PoC、培训、内部自动化
  • 低并发 PRD、文档、研发辅助
  • 验证 Hermes 的 Agent 能力与企业微信入口
  • API 始终位于本机或受控内网

不应直接这样做

  • 把 8000/8642 端口裸露到公网
  • 多人共用个人账号绕过授权或配额
  • 让企业微信任意成员执行终端工具
  • 把真实密钥写进 Git、截图或培训材料
嵌套 Agent 风险 Hermes 自身会规划和调用工具，而 Cursor CLI 也是 Agent。二者叠加可能出现工具协议兼容、延迟、重复规划或不可预期文件操作。生产化前必须完成“纯问答、流式、长文本、工具调用、超时、并发”六类测试；高稳定性场景优先改接受支持的正式模型 API。

5.4 配置为用户级 systemd 服务

先创建只读密钥文件 ~/.config/agent-cli-to-api.env，权限必须为 600：

cat > ~/.config/agent-cli-to-api.env <<'EOF'
CODEX_GATEWAY_TOKEN=替换为随机令牌
CODEX_WORKSPACE=/home/hermes/workspace/gateway-sandbox
CODEX_LOG_MODE=summary
CURSOR_AGENT_MODEL=auto
# 若不用浏览器登录态，再加入 CURSOR_API_KEY=...
EOF
chmod 600 ~/.config/agent-cli-to-api.env
cat > ~/.config/systemd/user/agent-cli-to-api.service <<'EOF'
[Unit]
Description=Cursor Agent OpenAI-compatible Gateway
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
WorkingDirectory=%h/apps/agent-cli-to-api
EnvironmentFile=%h/.config/agent-cli-to-api.env
ExecStart=%h/.local/bin/uv run agent-cli-to-api cursor-agent --host 127.0.0.1 --port 8000
Restart=on-failure
RestartSec=5
NoNewPrivileges=true
PrivateTmp=true
ProtectSystem=full

[Install]
WantedBy=default.target
EOF

systemctl --user daemon-reload
systemctl --user enable --now agent-cli-to-api
systemctl --user status agent-cli-to-api
journalctl --user -u agent-cli-to-api -f
培训演示点让学员观察请求从 /v1/chat/completions 进入、网关拉起 Cursor Agent、再返回标准 OpenAI JSON。明确网关负责“协议转换”，不负责业务权限。

Module 06安装并配置 Hermes Agent

安装 Hermes

# 官方安装器；生产环境可先下载审计脚本再执行
curl -fsSL https://hermes-agent.nousresearch.com/install.sh | bash
source ~/.bashrc

hermes --version
hermes setup

官方安装器会将代码放到 ~/.hermes/hermes-agent/，命令位于 ~/.local/bin/hermes。Linux 前置依赖主要是 Git、curl 和 xz-utils。

把模型指向 Cursor 转换网关

推荐运行交互式命令 hermes model，选择 Custom endpoint：

API Base URLhttp://127.0.0.1:8000/v1
API Key与 CODEX_GATEWAY_TOKEN 相同
Model namecursor:（固定 provider 时模型名可能被忽略）
API modechat_completions

也可检查 ~/.hermes/config.yaml，核心结构如下：

model:
  default: "cursor:"
  provider: custom
  base_url: "http://127.0.0.1:8000/v1"
  api_key: "替换为网关令牌"
  api_mode: chat_completions
版本注意Hermes 当前以 config.yaml 作为模型、provider 与 base URL 的权威配置。不要只设置旧式 OPENAI_BASE_URL；Base URL 要包含 /v1。

验证 Hermes 到模型网关的链路

hermes config show
hermes "只回答 HERMES_OK，不要调用工具"

6.1 开启 Hermes 自身的 Agent API

该接口让其他内网系统调用“完整 Hermes 能力”，而不是直接调用 Cursor 网关。

cat >> ~/.hermes/.env <<'EOF'
API_SERVER_ENABLED=true
API_SERVER_HOST=127.0.0.1
API_SERVER_PORT=8642
API_SERVER_KEY=替换为另一个独立随机令牌
EOF

另开终端：

export HERMES_API_KEY='替换为 Hermes API 令牌'
curl -sS http://127.0.0.1:8642/health

  -H "Authorization: Bearer $HERMES_API_KEY" \

    "model":"hermes-agent",

Module 07企业微信机器人：推荐原生 WebSocket

推荐路径已经内置 Hermes 原生支持 WeCom AI Bot WebSocket 网关：实时双向通信、无需公网 IP、无需自己处理 AES 回调加解密，比“自建一个 Webhook 转发服务”更简单也更安全。
企业微信只负责消息通道，Agent 的权限、工具、Skill 和会话由 Hermes 控制。

7.1 创建并连接 AI Bot

交互式配置（推荐）

hermes gateway setup
# 选择 WeCom
# 用企业微信手机端扫描终端二维码
# 配置用户白名单、群白名单和通知主频道

若扫码流程不可用：在企业微信管理后台进入“应用管理 → 创建应用 → AI Bot”，复制 Bot ID 与 Secret，再回到向导手动输入。

安全优先的手动配置

cat >> ~/.hermes/.env <<'EOF'
WECOM_BOT_ID=替换为BotID

7.2 三种“企业微信机器人”不要混淆

方案能否收消息公网要求本方案建议
群消息推送 Webhook（原群机器人）不能，仅主动推送无入站公网只用于发布成功通知
WeCom AI Bot WebSocket能，双向无需固定公网 IP首选，Hermes 原生支持
自建应用 Callback能，双向需要公网 HTTPS 回调有复杂企业权限或传统应用需求时使用
如只需发布通知PRD 发布脚本可 POST 到群消息推送 Webhook；但它不能接收“帮我写 PRD”这样的提问。单个消息推送每分钟不超过 20 条，Markdown 内容最长 4096 字节（以企业微信最新官方文档为准）。

Module 09建立“PRD 一键发布”Hermes Skill

Skill 不是一段万能提示词，而是一套可重复执行的操作规程。它告诉 Hermes：何时触发、写到哪里、如何校验、何时必须征得用户确认、怎样发布并验收。

9.1 创建 Skill 目录

mkdir -p ~/.hermes/skills/product/publish-prd/{references,templates,scripts}

9.2 SKILL.md 示例

---
name: publish-prd
description: Use when drafting, validating, versioning, and publishing a PRD to the shared Git Page Docs website.
version: 1.0.0
author: Enterprise AI Team
platforms: [linux]
metadata:
  hermes:
    tags: [prd, product, documentation, github-pages]
    category: product
    requires_toolsets: [terminal]
required_environment_variables:
  - name: PRD_REPO
    prompt: Absolute path of the PRD repository
    required_for: Writing and publishing PRDs
---
# Publish PRD

## When to Use
Use this skill when the user asks to create, update, review, or publish a PRD.

## Procedure
1. Clarify product goal, target users, scope, non-goals, constraints, and acceptance criteria.
2. Read `${HERMES_SKILL_DIR}/templates/prd-template.md`.
3. Read the generated Git Page Docs config and identify the active version and Chinese language directory.
4. Draft the PRD under `$PRD_REPO/gitpagedocs/docs/versions/<active-version>/<language>/<project>.md`.
5. Never invent business decisions. Mark unresolved decisions as `待确认`.
6. Run `${HERMES_SKILL_DIR}/scripts/validate-prd.sh <file>`.
7. Show the user: file path, summary, open questions, and planned commit message.
8. Ask for explicit confirmation before any `git commit` or `git push`.
9. After approval, run:
   `${HERMES_SKILL_DIR}/scripts/publish-prd.sh <file>`
10. Return the commit SHA, Actions run URL, and final Pages URL.

## Safety Rules
- Never print or commit tokens, cookies, webhook URLs, customer secrets, or personal data.
- Never force-push or rewrite history.
- Stop if the working tree contains unrelated changes.
- Publishing is an external write and always requires explicit user confirmation.
- If GitHub Pages deployment fails, report the failure; do not claim success.

## Pitfalls
- GitHub Pages may take several minutes after the first deployment.
- `gitpagedocs` structure can change between versions; pin and test the CLI version.
- A public GitHub Pages site is not suitable for confidential PRDs.

## Verification
1. `git status --short` contains no unexpected files.
2. The validation script exits 0.
3. The push succeeds and returns a commit SHA.
4. GitHub Actions deployment concludes successfully.
5. The final PRD URL returns HTTP 200.

9.3 校验脚本

cat > ~/.hermes/skills/product/publish-prd/scripts/validate-prd.sh <<'EOF'
#!/usr/bin/env bash
set -euo pipefail

file="${1:?用法: validate-prd.sh PRD_FILE}"
[[ -f "$file" ]] || { echo "文件不存在: $file" >&2; exit 2; }

required=(
  "## 1. 文档信息"
  "## 2. 背景与问题"
  "## 3. 目标与非目标"
  "## 8. 验收标准"
  "## 9. 风险、依赖与开放问题"
)

for heading in "${required[@]}"; do
  grep -Fq "$heading" "$file" || {
    echo "缺少必填章节: $heading" >&2
    exit 3
  }
done

if grep -Eqi '(sk-[A-Za-z0-9_-]{16,}|CURSOR_API_KEY=|WECOM_SECRET=|webhook/send\?key=)' "$file"; then
  echo "疑似包含密钥或 Webhook，拒绝发布" >&2
  exit 4
fi

echo "PRD_VALID: $file"
EOF
chmod +x ~/.hermes/skills/product/publish-prd/scripts/validate-prd.sh

9.4 发布脚本

cat > ~/.hermes/skills/product/publish-prd/scripts/publish-prd.sh <<'EOF'
#!/usr/bin/env bash
set -euo pipefail

file="${1:?用法: publish-prd.sh PRD_FILE}"
repo="${PRD_REPO:?必须设置 PRD_REPO}"
branch="${PRD_BRANCH:-main}"

cd "$repo"
git rev-parse --is-inside-work-tree >/dev/null
file_rel="$(realpath --relative-to="$repo" "$file")"

# 拒绝夹带无关改动
unexpected="$(git status --porcelain | awk '{print $2}' | grep -vFx "$file_rel" || true)"
if [[ -n "$unexpected" ]]; then
  echo "检测到无关改动，停止发布：" >&2
  echo "$unexpected" >&2
  exit 10
fi

"$HOME/.hermes/skills/product/publish-prd/scripts/validate-prd.sh" "$file_rel"

git add -- "$file_rel"
git diff --cached --quiet && { echo "没有可发布的变更"; exit 0; }

title="$(grep -m1 '^# ' "$file" | sed 's/^# //')"
git commit -m "docs(prd): publish ${title:-PRD}"
git push origin "HEAD:$branch"

sha="$(git rev-parse HEAD)"
remote="$(git remote get-url origin)"
echo "PUBLISHED_COMMIT=$sha"
echo "REMOTE=$remote"
echo "等待 GitHub Actions 完成后再确认 Pages 链接。"
EOF
chmod +x ~/.hermes/skills/product/publish-prd/scripts/publish-prd.sh

Module 12常见故障排查

Cursor CLI 显示 invalid API key

先运行 agent status，再检查 curl -v https://api2.cursor.sh。这类报错也可能来自 DNS、TLS、VPN 或公司代理。确认使用的是 User API Key，不是 Admin API Key，并更新 CLI。

网关 /healthz 正常，但聊天请求失败

依次验证：Cursor CLI 独立问答 → 网关 token → CODEX_WORKSPACE 权限 → systemd 环境变量 → journalctl --user -u agent-cli-to-api。固定 provider 启动时，客户端的 model 字符串可能被忽略。

Hermes 请求到了 /chat/completions，缺少 /v1

检查 ~/.hermes/config.yaml 的 model.base_url 是否完整写为 http://127.0.0.1:8000/v1。重新运行 hermes model 的 Custom Endpoint 向导。

企业微信机器人不回复

检查 Bot ID/Secret、用户或群白名单、到 openws.work.weixin.qq.com 的网络、aiohttp/httpx 依赖及 Hermes Gateway 日志。不要用“群消息推送 Webhook”期待接收消息。

PRD 推送成功但网站没更新

检查 Actions 是否成功、Pages Source 是否为 GitHub Actions、站点 base path 是否包含仓库名、gitpagedocs/config.json 的 rendering URL 是否正确。首次发布通常有延迟。

Agent 误提交了其他文件

发布脚本应在提交前检查工作树并拒绝无关变更。用 git show --stat HEAD 审核；不要 force-push。已发布错误内容应使用新的反向提交回滚并保留审计历史。
