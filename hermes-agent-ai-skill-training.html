边界先讲清楚

Cursor 转 API 是社区桥接，不是 Cursor 官方模型 API。
上线前必须核对 Cursor 当前服务条款、企业订阅策略、自动化调用与共享账号规则。禁止将个人登录态做成公共 API，也不要承诺“无限量/零成本”。
GitHub Pages 的网页可能公开可访问。
即使源仓库为私有，Pages 的可见性仍取决于组织套餐与设置。含客户信息、商业机密、密钥的 PRD 应使用 GitHub Enterprise 可控 Pages 或内网自托管。

03 / INSTALLATION

Ubuntu 环境部署操作手册

以下命令以专用用户 hermes 为例。不要用 root 登录 Cursor，也不要让多个 Linux 用户混用同一份认证目录。

准备主机和专用账号

建议 PoC 至少 2 vCPU / 4 GB RAM / 30 GB 磁盘，Ubuntu 22.04 或 24.04 LTS。先更新系统并创建低权限账号。

sudo apt update && sudo apt upgrade -y
sudo apt install -y git curl xz-utils build-essential ca-certificates jq
sudo adduser --disabled-password --gecos "" hermes
sudo install -d -o hermes -g hermes /srv/prd-site
sudo -iu hermes
生产环境建议启用磁盘加密、自动安全更新、主机 EDR，并限制该账号的 sudo 权限。Hermes 可执行工具命令，权限边界就是风险边界。

安装 Cursor CLI 并完成账号登录

必须在未来运行 systemd 服务的同一个 hermes 用户下登录，以便服务读取同一份认证状态。

curl https://cursor.com/install -fsS | bash
echo 'export PATH="$HOME/.local/bin:$PATH"' >> ~/.bashrc
source ~/.bashrc

agent --version
agent login
agent status
agent models

无桌面服务器可按 CLI 提示复制登录链接到个人浏览器完成认证。若企业禁止交互式 OAuth，请改用企业批准的官方 API 供应商，不要绕过策略。

检查点：agent status 显示已登录，并且执行一次测试问答能正常返回后再进入下一步。

安装 uv 与 agent-cli-to-api

社区项目当前要求 Python 3.10+，推荐使用 uv。培训环境建议锁定经过验证的 commit，不要让生产服务每天追踪 main。

curl -LsSf https://astral.sh/uv/install.sh | sh
source ~/.bashrc
cd ~
git clone https://github.com/leeguooooo/agent-cli-to-api.git
cd agent-cli-to-api
git rev-parse HEAD | tee DEPLOYED_COMMIT
uv sync

# 诊断 CLI 和依赖
uv run agent-cli-to-api doctor

供应链建议 建立内部 fork；审查依赖；保存 commit SHA；升级前在测试机回归；不要直接把远程脚本作为 root 执行。

生成网关令牌并前台试运行

GATEWAY_TOKEN="$(openssl rand -hex 32)"
umask 077
printf 'CODEX_GATEWAY_TOKEN=%s\nCODEX_LOG_MODE=summary\n' "$GATEWAY_TOKEN" \
  > ~/agent-cli-to-api/gateway.env

cd ~/agent-cli-to-api
set -a; source gateway.env; set +a
uv run agent-cli-to-api cursor-agent --host 127.0.0.1 --port 8000

另开一个终端执行：

sudo -iu hermes
TOKEN="$(cut -d= -f2 ~/agent-cli-to-api/gateway.env | head -1)"
curl -fsS http://127.0.0.1:8000/healthz
curl -fsS http://127.0.0.1:8000/v1/models \
  -H "Authorization: Bearer $TOKEN" | jq
curl -fsS http://127.0.0.1:8000/v1/chat/completions \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"model":"auto","stream":false,"messages":[{"role":"user","content":"只回复 API_OK"}]}' | jq

用 systemd 托管转换网关

退出到管理员账号，创建服务。HOME 和 PATH 必须正确，否则服务找不到 Cursor CLI 或登录态。

# /etc/systemd/system/agent-cli-to-api.service
[Unit]
Description=Cursor CLI to OpenAI-compatible API
After=network-online.target
Wants=network-online.target

[Service]
Type=simple
User=hermes
Group=hermes
WorkingDirectory=/home/hermes/agent-cli-to-api
Environment=HOME=/home/hermes
Environment=PATH=/home/hermes/.local/bin:/usr/local/sbin:/usr/local/bin:/usr/sbin:/usr/bin
EnvironmentFile=/home/hermes/agent-cli-to-api/gateway.env
ExecStart=/home/hermes/agent-cli-to-api/.venv/bin/agent-cli-to-api cursor-agent --host 127.0.0.1 --port 8000
Restart=on-failure
RestartSec=5
NoNewPrivileges=true
PrivateTmp=true
ProtectSystem=strict
ReadWritePaths=/home/hermes

[Install]
WantedBy=multi-user.target
sudo chmod 600 /home/hermes/agent-cli-to-api/gateway.env
sudo systemctl daemon-reload
sudo systemctl enable --now agent-cli-to-api
sudo systemctl status agent-cli-to-api --no-pager
sudo journalctl -u agent-cli-to-api -n 100 --no-pager

安装 Hermes Agent

切回 hermes 用户。官方安装器会处理 Python、Node.js、ripgrep、ffmpeg 等依赖。

sudo -iu hermes
curl -fsSL https://hermes-agent.nousresearch.com/install.sh | bash
source ~/.bashrc
hermes doctor
hermes setup

如服务器不需要浏览器自动化，可使用：

curl -fsSL https://hermes-agent.nousresearch.com/install.sh \
  | bash -s -- --skip-browser

让 Hermes 使用本机 Cursor 转换 API

推荐运行 hermes model，选择 Custom / OpenAI-compatible endpoint，或核对以下配置。模型和 URL 放 config.yaml，令牌放权限为 600 的 .env。

# ~/.hermes/config.yaml（合并到现有 model 段，不要覆盖其他配置）
model:
  default: auto
  provider: custom
  base_url: http://127.0.0.1:8000/v1
# 把同一个 gateway token 写入 Hermes secret 文件
TOKEN="$(cut -d= -f2 ~/agent-cli-to-api/gateway.env | head -1)"
umask 077
printf '\nOPENAI_API_KEY=%s\n' "$TOKEN" >> ~/.hermes/.env
chmod 600 ~/.hermes/.env ~/.hermes/config.yaml

hermes config check
hermes doctor
hermes
不要重复添加多个 model: 顶层键。先备份 cp ~/.hermes/config.yaml ~/.hermes/config.yaml.bak，再合并配置。不同 Hermes 版本的向导字段可能变化，以当期官方文档与 hermes model 为准。
04 / WECOM

部署企业微信双向 AI 机器人

Hermes 已原生支持企业微信 AI Bot WebSocket 网关。它不是传统“群机器人 webhook”：WebSocket 方案能收发消息、无需公网入站端口，最适合本方案。

WSS

双向实时

连接 wss://openws.work.weixin.qq.com，断线自动重连。

0

公网入站端口

AI Bot 模式只需出站网络，不必暴露回调服务。

ACL

访问控制

私聊、群组和群内用户均可配置 allowlist。

创建并绑定企业微信 AI Bot

  1. 在 Hermes 主机执行 hermes gateway setup。
  2. 选择 WeCom（企业微信）。
  3. 推荐用企业微信手机端扫描终端二维码；无法扫码时，到管理后台创建 AI Bot，手工填入 Bot ID 与 Secret。
  4. 配置通知主频道，并选择白名单策略。
hermes gateway setup
# 完成后检查密钥文件权限
chmod 600 ~/.hermes/.env ~/.hermes/config.yaml

生产环境不要使用 open 策略

# ~/.hermes/config.yaml 示例片段
platforms:
  wecom:
    enabled: true
    extra:
      dm_policy: allowlist
      group_policy: allowlist
      allow_from:
        - user_product_owner
        - user_admin
      group_allow_from:
        - prd_pilot_group_id
      groups:
        prd_pilot_group_id:
          allow_from:
            - user_product_owner
            - user_product_manager

Bot ID、Secret 建议由向导写入 ~/.hermes/.env，不要提交到 Git。

启动并验证 Gateway

hermes gateway
# 在企业微信向机器人发送：
# “只回复 WECOM_OK，不要调用工具”

再发送：“生成一份员工请假流程 PRD 草稿，保存到 /srv/prd-site/drafts/leave-v1.md，暂不发布。”确认能写入文件后再启用发布 Skill。

托管 Hermes Gateway

[Unit]

[Service]

PrivateTmp=true

[Install]

05 / PRD SKILL

建立 PRD 共享站与“一键发布”Skill

选用开源 Git Page Docs 管理版本化 Markdown，并通过 GitHub Actions 发布到 GitHub Pages。Skill 负责把自然语言动作收敛为一条受控 SOP。

5.1 初始化共享站

创建 GitHub 仓库并配置凭据

建议创建组织仓库 prd-knowledge-base。敏感 PRD 使用私有仓库，并确认 Pages 的访问策略满足企业要求。

sudo -iu hermes
cd /srv/prd-site
git init
git branch -M main
git config user.name "Hermes PRD Bot"
git config user.email "hermes-bot@your-company.example"

# 推荐使用机器人账号的 SSH Deploy Key（仅该仓库写权限）
ssh-keygen -t ed25519 -f ~/.ssh/prd_publish -C "hermes-prd-publisher"
cat ~/.ssh/prd_publish.pub
# 将公钥添加到 GitHub 仓库 Settings → Deploy keys，并按需允许写入
# ~/.ssh/config
Host github-prd
  HostName github.com
  User git
  IdentityFile ~/.ssh/prd_publish
  IdentitiesOnly yes
chmod 600 ~/.ssh/config ~/.ssh/prd_publish
git remote add origin git@github-prd:YOUR_ORG/prd-knowledge-base.git

生成 Git Page Docs 目录

cd /srv/prd-site
node --version   # 需 Node.js 18+，推荐 20+
npx @gitpagedocs/cli

# 生成本地布局，降低运行期依赖官方模板的风险
npx gitpagedocs --layoutconfig
git add .
git commit -m "chore: initialize PRD knowledge base"
git push -u origin main

典型目录：

gitpagedocs/
├── config.json
├── layouts/
└── docs/versions/
    └── 1.0.0/
        ├── config.json
        └── zh/
            ├── index.md
            └── employee-leave.md

启用 GitHub Pages

仓库 Settings → Pages → Source 选择 GitHub Actions。也可使用 CLI 一次生成 workflow、提交并推送：

cd /srv/prd-site
npx gitpagedocs --push \
  --owner YOUR_ORG \
  --repo prd-knowledge-base

发布地址通常为：

https://YOUR_ORG.github.io/prd-knowledge-base/v/1.0.0/?lang=zh
首次使用前应审查 CLI 自动生成的 .github/workflows/gitpagedocs-pages.yml。把 Actions 依赖固定到 commit SHA，最小化 GITHUB_TOKEN 权限。

5.3 编写 SKILL.md

---
name: prd-publisher
description: Use when the user asks to create, validate, version, or publish a PRD to the shared documentation site.
version: 1.0.0
author: Your Company
platforms: [linux]
metadata:
  hermes:
    tags: [product, prd, publishing, git]
---
# PRD Publisher

## When to use
Use this skill when a user asks to “编写 PRD”“发布 PRD”“更新共享文档”
or “把需求文档发到网站”.

## Required inputs
- PRD title and owner
- source Markdown path under `/srv/prd-site/drafts/`
- semantic version, default `1.0.0`
- visibility: `public` or `internal`

## Safety gates
1. Never publish secrets, access tokens, webhook URLs, credentials, personal IDs,
   customer data, unreleased financial data, or documents marked confidential.
2. Publishing requires an explicit user confirmation containing “确认发布”.
3. Refuse paths outside `/srv/prd-site/drafts/`.
4. Do not bypass failed validation, failed tests, failed push, or failed Pages build.
5. Never use `git push --force`.

## Procedure
1. Draft from `templates/prd-template.md`.
2. Verify required sections: background, goals, non-goals, personas, flows,
   requirements, acceptance criteria, metrics, risks, rollout and rollback.
3. Scan for secrets and sensitive fields. If found, stop and report locations.
4. Present title, version, target URL and diff summary; request “确认发布”.
5. After confirmation run:
   `scripts/publish-prd.sh SOURCE VERSION SLUG`
6. Parse the final `PUBLISHED_URL=...` line.
7. Send a WeCom reply with title, version, commit SHA and URL.

## Verification
- script exit code is 0
- remote commit SHA equals local HEAD
- published URL returns HTTP 200 within the configured timeout
- reply contains no credential or webhook secret

## Rollback
Create a normal revert commit with `git revert <sha>`, push it, wait for Pages,
then post a rollback notice. Never rewrite shared history.

5.4 一键发布脚本

以下是培训版基线。Git Page Docs 的版本配置文件需要登记页面路由，实际项目应在脚本中增加 JSON 路由更新，或使用其 gitpagedocs document:file 命令生成并挂载页面。

#!/usr/bin/env bash
set -Eeuo pipefail

SOURCE="${1:?用法: publish-prd.sh SOURCE VERSION SLUG}"
VERSION="${2:?缺少 VERSION，例如 1.0.0}"
SLUG="${3:?缺少 SLUG，例如 employee-leave}"
REPO="/srv/prd-site"
SOURCE_REAL="$(realpath "$SOURCE")"
DRAFT_ROOT="$(realpath "$REPO/drafts")"
TARGET_DIR="$REPO/gitpagedocs/docs/versions/$VERSION/zh"
TARGET="$TARGET_DIR/$SLUG.md"
SITE="https://YOUR_ORG.github.io/prd-knowledge-base/v/$VERSION/?lang=zh&name=$SLUG"

[[ "$SOURCE_REAL" == "$DRAFT_ROOT/"* ]] || {
  echo "拒绝：源文件必须位于 $DRAFT_ROOT" >&2; exit 2;
}
[[ "$VERSION" =~ ^[0-9]+\.[0-9]+\.[0-9]+$ ]] || {
  echo "拒绝：VERSION 必须是语义化版本" >&2; exit 2;
}
[[ "$SLUG" =~ ^[a-z0-9][a-z0-9-]{1,62}$ ]] || {
  echo "拒绝：SLUG 只能包含小写字母、数字和连字符" >&2; exit 2;
}

# 基础质量门禁
for heading in "背景" "目标" "功能需求" "验收标准" "风险" "回滚"; do
  grep -q "$heading" "$SOURCE_REAL" || {
    echo "校验失败：缺少章节 $heading" >&2; exit 3;
  }
done

# 凭据和高风险字段扫描；企业应接入 gitleaks + DLP
if grep -Eni \
  '(BEGIN (RSA|OPENSSH) PRIVATE KEY|api[_-]?key\s*[:=]|secret\s*[:=]|password\s*[:=]|qyapi\.weixin\.qq\.com/cgi-bin/webhook/send\?key=)' \
  "$SOURCE_REAL"; then
  echo "校验失败：疑似包含密钥或 webhook" >&2; exit 4
fi

mkdir -p "$TARGET_DIR"
install -m 0644 "$SOURCE_REAL" "$TARGET"

cd "$REPO"
git pull --ff-only origin main
git add "$TARGET" gitpagedocs/
git diff --cached --quiet && {
  echo "没有内容变化，无需发布"; exit 0;
}
git commit -m "docs(prd): publish $SLUG v$VERSION"
git push origin main

SHA="$(git rev-parse --short HEAD)"
REMOTE_SHA="$(git ls-remote origin refs/heads/main | cut -c1-7)"
[[ "$SHA" == "$REMOTE_SHA" ]] || {
  echo "校验失败：远端 SHA 与本地不一致" >&2; exit 5;
}

echo "COMMIT_SHA=$SHA"
echo "PUBLISHED_URL=$SITE"

5.5 使用者的一句话工作流

先编写，不发布

“为采购审批系统编写 v1.0 PRD，负责人张三，保存为草稿；补齐用户流程、验收标准和回滚方案，暂不发布。”

人工确认后发布

“检查采购审批 PRD 的敏感信息和必填章节，展示变更摘要。目标版本 1.0.0，slug 为 procurement-approval。确认发布。”

理想回执：“已发布《采购审批系统 PRD》v1.0.0；Commit a1b2c3d；访问地址 …；质量门禁全部通过。”

学员练习

  1. 完成主机、CLI、API 三层健康检查，并截图保留结果。
  2. 把企微策略从 open 改成白名单，验证未授权用户无法触发。

  5. 执行一次非强推回滚，并验证网站内容恢复。
07 / SECURITY

安全、合规与生产化要求

A

账号

  • 独立服务账号
  • 禁止共享个人账号
  • 定期复核订阅条款
  • 人员离职立即撤权
N

网络

  • API 仅监听 127.0.0.1
  • 不做公网反向代理
  • 限制主机出站域名
  • Callback 必须 HTTPS
D

数据

  • PRD 分级分类
  • 发布前 DLP 扫描
